Progressive Search — How Results Appear

0ms Spotlight bar opens (clean, blank) <50ms Keyword matches appear (SQLite FTS5, BM25 ranking) ~300ms Semantic results stream in (local embeddings via sqlite-vec) ~500ms Activity-ranked results blend in (recency + frequency + co-access) Scoring formula: score = (semantic_score × 0.5) + (keyword_score × 0.3) + (activity_score × 0.2) WebSocket streaming: WS /api/v1/search/stream Client sends: { "query": "dentist receipt", "limit": 20 } Server sends: { "phase": "keyword", "results": [...], "timing_ms": 12 } Server sends: { "phase": "semantic", "results": [...], "timing_ms": 280 } Server sends: { "phase": "activity", "results": [...], "timing_ms": 450 } Server sends: { "phase": "done", "total": 47 }

Background Intelligence — Silently Improves Search

Version Chains
Detection: Levenshtein distance on filenames + SHA-256 content hash similarity.

report_v1.docx → report_v2.docx → report_final.docx

Stored in file_relationships with relationship='version_chain'
Topic Clusters
Detection: k-means clustering on embedding vectors. Auto-labels clusters via most common keywords.

Tax returns + W-2s + bank statements → "Tax 2025"

Stored with relationship='topic_cluster'
Temporal Bundles
Detection: group files modified within configurable window (default: 1 hour).

"Everything from last Tuesday's meeting"

Stored with relationship='temporal_bundle'
Proactive Alerts
Duplicate scan: group by content_hash
Stale files: not accessed in >90 days
Large unused: >50MB, not accessed in >30 days
Dead code: Knip (JS/TS), Vulture (Python)

Background job every 6 hours
